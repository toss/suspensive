# 동기

React에는 이미 훌륭한 기본 요소들이 있습니다 — Suspense, ErrorBoundary, lazy. 하지만 실제 애플리케이션에서 사용하면 빈틈이 드러납니다. @suspensive/react는 그 빈틈을 채웁니다.

### Suspense 바운더리에서 SSR을 건너뛰고 싶을 때

React의 `<Suspense>`는 SSR과 스트리밍과 함께 동작합니다. 그래도 바운더리를 **클라이언트에서만** 렌더링하고 싶을 때가 있습니다 — 서버에서는 `fallback`만 보여 주고, 하이드레이션 이후에 `children`을 마운트하는 경우(클라이언트 전용 API·위젯, 점진적 마이그레이션 등). React에는 이를 위한 내장 prop이 없어 `dynamic(..., { ssr: false })`나 수동 가드를 쓰는 경우가 많습니다.

Suspensive의 [`<Suspense clientOnly/>`](/docs/react/Suspense)는 `clientOnly` prop 하나로 그 의도를 선언할 수 있습니다 — 별도 래퍼가 필요 없습니다.

### ErrorBoundary가 너무 복잡합니다

`react-error-boundary`는 세 가지 fallback 인터페이스를 제공합니다. Suspensive의 [`<ErrorBoundary/>`](/docs/react/ErrorBoundary)를 활용하면 하나의 `fallback` prop으로 통합하고, `shouldCatch`로 처리할 에러를 선택할 수 있습니다.

### ErrorBoundary 초기화가 번거롭습니다

여러 `<ErrorBoundary/>`의 `resetKeys`를 각각 관리하는 것은 고됩니다. Suspensive의 [`<ErrorBoundaryGroup/>`](/docs/react/ErrorBoundaryGroup)을 활용하면 한 번의 호출로 모두 초기화할 수 있습니다.

### 로딩 스피너가 깜빡입니다

50ms만 보이는 스피너는 없느니만 못합니다. Suspensive의 [`<Delay/>`](/docs/react/Delay)를 활용하면 짧은 로딩 상태를 숨겨서, 진짜 기다려야 할 때만 스피너를 보여줄 수 있습니다.

### 클라이언트 전용 코드가 hydration 불일치를 일으킵니다

Suspensive의 [`<ClientOnly/>`](/docs/react/ClientOnly)와 [`useIsClient`](/docs/react/useIsClient)를 활용하면 안전하게 클라이언트에서만 렌더링할 수 있습니다 — 코드 곳곳에 `typeof window !== 'undefined'`를 흩뿌릴 필요가 없습니다.

### 같은 props를 반복 전달하고 있습니다

모든 `<Suspense/>`에 같은 `fallback`을 넘기지 마세요. Suspensive의 [`<DefaultPropsProvider/>`](/docs/react/DefaultPropsProvider)를 활용하면 한 번만 설정하면 됩니다.

### `React.lazy`는 사전 로딩도 에러 복구도 안 됩니다

Suspensive의 [`lazy`](/docs/react/lazy)를 활용하면 `.load()`로 사전 로딩하고, 청크 로딩 실패 시 자동 재시도할 수 있습니다 — 프로덕션에서야 비로소 필요하다는 걸 알게 되는 기능입니다.
